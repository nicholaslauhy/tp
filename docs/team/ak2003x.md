# Khoo Junhao - Project Portfolio Page

## Project: FinTrackPro

FinTrackPro is a CLI-based individual BTO budget planner for university students planning to
apply for BTO. It turns messy finances into a clear downpayment plan, showing how much the
individual needs to save and whether additional financing is required.

### Enhancements Implemented

**1. Profile Component (`Profile.java`)**
- **What it does:** Central data model storing all user financial state: `monthlyAllowance`,
  `currentSavings`, `contributionRatio`, `housePrice`, `btoGoal`, `deadline`, and `currentMonth`.
  Calling `setContributionRatio()` or updating `housePrice` automatically triggers
  recalculation of `btoGoal` via `BtoCalculator`.
- **Justification:** A single source of truth for all financial data is essential for
  ensuring that downstream components (SummaryReport, CommandHandler, Storage) remain
  consistent and do not recompute state independently.
- **Highlights:** All financial fields use `BigDecimal` for precision. Defensive assertions
  guard against null inputs and negative values, and the ratio is constrained to [0.01, 1.0].
  The auto-recalculation of `btoGoal` on every relevant setter means the profile never holds
  stale goal data.

**2. `allowance` and `ratio` Commands**
- **What it does:** `allowance` lets the user update their monthly allowance interactively,
  displaying the current value before prompting for a new one. `ratio` does the same for
  the BTO contribution ratio, and automatically recalculates `btoGoal` on update.
- **Justification:** Students' financial circumstances change over time; without a way to
  update these values, the BTO readiness report would become increasingly inaccurate.
- **Highlights:** Both commands display the current value before the prompt so users can
  make an informed decision without needing to run `summary` first.

**3. `savings` Command**
- **What it does:** Adds a specified surplus amount to the user's existing savings balance,
  displaying the new total after each transaction.
- **Justification:** Savings accumulate incrementally; overwriting the balance each time
  would be error-prone. An additive model better reflects real-world saving behaviour.
- **Highlights:** Uses `BigDecimal.add()` for precision and validates input via
  `InputUtil.readMoney()` to reject negative or malformed amounts.

**4. `InputUtil` — `readMoney()` fix and `readRatio()` implementation**
- **What it does:** Fixed `readMoney()` by removing a hardcoded `0.025` multiplier that
  incorrectly scaled all monetary inputs. Added `readRatio()` with regex validation and
  a [0.01, 1.0] range check.
- **Justification:** The hardcoded multiplier caused all salary and savings inputs to be
  silently scaled down, corrupting the profile state. Separating ratio input into its own
  validator prevents invalid values from entering the system.
- **Highlights:** `readRatio()` uses a regex pattern to enforce at most 2 decimal places,
  consistent with the precision constraints used across the app.

**5. Assertions and Logging (`Profile`, `SummaryReport`, `Ui`)**
- **What it does:** Added `assert` statements throughout `Profile`, `SummaryReport`, and
  `Ui` to catch invariant violations at runtime. Integrated `LoggerUtil` into all three
  classes for structured file-based logging.
- **Justification:** Defensive programming guards against subtle bugs introduced during
  future refactors, and structured logging gives visibility into runtime state without
  cluttering console output.
- **Highlights:** `Profile` assertions prevent null names, null deadlines, negative
  allowances, and out-of-range ratios from corrupting the data model silently.

**6. JUnit Tests (`ProfileTest`, `CommandHandlerTest`)**
- **What it does:** `ProfileTest` covers happy and sad paths for all financial
  setters/getters including `monthlyAllowance`, `currentSavings`, `contributionRatio`,
  `housePrice`, and the `btoGoal` auto-recalculation chain. `CommandHandlerTest` covers
  input validation and `btoGoal` recalculation triggered via commands.
- **Justification:** The Profile class is the most stateful component in the app;
  thorough unit tests catch regressions when any financial field or setter logic changes.
- **Highlights:** Tests verify that `setContributionRatio()` and `setHousePrice()` trigger
  correct `btoGoal` recalculation end-to-end through `BtoCalculator`, not just in isolation.
